### Clock Skew Summary

- **Clock skew** is the difference in time between two systems that should have synchronized clocks.
- It can cause:
    - Authentication failures (JWT, AWS signed requests)
    - SSL/TLS certificate validation errors
    - Incorrect log timestamps
    - Issues in distributed systems and database replication

**AWS Example:**

```
aws s3 ls
```

Error:

```
RequestTimeTooSkewed
```

This happens when your system's clock differs significantly from AWS's clock.

### How it's prevented

- Systems use **NTP (Network Time Protocol)** to synchronize time.
- On Linux, **`chronyd`** or **`systemd-timesyncd`** runs in the background and periodically syncs the system clock.
- In AWS, EC2 instances typically use the **Amazon Time Sync Service**, so no separate NTP server needs to be configured.

### Do we have to keep `chronyd` running?

**Yes.** But:

- You don't manually start it every day.
- On most AWS AMIs, it's already installed and enabled.
- Verify it's running after provisioning a new server:
    
    ```
    systemctl status chronyd
    timedatectl status
    ```
    
- In production, monitoring tools alert you if `chronyd` stops. Applications **do not** check `chronyd` before every request—they simply use the operating system's clock.

### Interview answer (30 seconds)

> Clock skew is the time difference between two systems. It can cause authentication failures, SSL certificate issues, incorrect logs, and distributed system inconsistencies. It is prevented by synchronizing system clocks using NTP. On Linux, services like `chronyd` keep the clock synchronized in the background. In AWS, EC2 instances typically use the Amazon Time Sync Service, so clock skew is rare unless time synchronization is disabled or misconfigured.

This is exactly the right question. The key misunderstanding is that **TLS does not compare your clock with the certificate issuer's current clock**.

It compares **your current time** with the **fixed validity timestamps embedded in the certificate**.

For example, suppose a Certificate Authority issues a certificate:

```
Valid From : 10 Jul 2026 00:00 UTC
Valid Until: 10 Jul 2027 23:59 UTC
```

These dates are **written into the certificate** when it is created.

Now imagine:

- The Certificate Authority's current time is 11:00 UTC.
- Your server's current time is 11:00 UTC ± a few milliseconds.

That's perfectly fine. They don't have to match exactly.

The problem is only when **your clock is significantly wrong**.

### Example

Certificate contains:

```
Valid From : 10 Jul 2026
```

Your server's clock says:

```
08 Jul 2026
```

Your server checks:

> "Is today after 10 Jul 2026?"

It answers:

> "No."

So it rejects the certificate as **not yet valid**.

Notice that the Certificate Authority's current time is irrelevant. Your server never asks:

> "CA, what time is it right now?"

It simply reads the dates inside the certificate.

### Why different time zones don't matter

Certificates use **UTC (Coordinated Universal Time)**.

Suppose you're in India:

```
IST = UTC + 5:30
```

Certificate:

```
Valid From: 10 Jul 2026 00:00 UTC
```

Your system converts your local time to UTC internally before comparing.

For example:

```
Your local time:
10 Jul 2026 05:30 IST

Equivalent UTC:
10 Jul 2026 00:00 UTC
```

The comparison succeeds.

So **time zones are not the issue**. Operating systems handle the conversion automatically.

### The real issue

The issue is **incorrect time**, not **different time zones**.

For example:

- ✔️ Your server: 10:00:05 UTC
- ✔️ Certificate timestamps: UTC
- ✔️ CA server: 10:00:01 UTC

No problem.

But if your server thinks it's:

- ❌ 8 Jul instead of 10 Jul
- ❌ 2028 instead of 2026

then certificate validation fails.

### Interview answer

> TLS certificates contain fixed `Not Before` and `Not After` timestamps in UTC. During the TLS handshake, the client compares those timestamps with its own system clock. It does not compare its clock with the Certificate Authority's live clock. If the client's clock is significantly incorrect, a valid certificate may be considered "not yet valid" or "expired," causing the TLS handshake to fail.

what is about ntp server time?

The NTP server's time is **the reference (accurate) time**. Every system synchronizes its clock with the NTP server, not with each other.

Think of it like an exam hall.

- The wall clock = **NTP server**
- Student A's watch = **Server A**
- Student B's watch = **Server B**

Nobody compares watches with each other. Everyone adjusts their watch according to the wall clock.

```
             NTP Server
          (10:00:00 UTC)
            /         \
           /           \
      Server A      Server B
      10:00:01      09:59:59
```

When `chronyd` runs on Server A:

1. It contacts the NTP server.
2. It asks, "What time is it?"
3. The NTP server replies, "10:00:00 UTC."
4. `chronyd` adjusts Server A's clock if necessary.

The same happens on Server B.

### Why not compare Server A with Server B?

Suppose:

- Server A is wrong.
- Server B is also wrong.

If they synchronized with each other, they could both stay wrong forever.

Instead, they synchronize with a trusted source.

### In AWS

The NTP server is the **Amazon Time Sync Service**.

```
Amazon Time Sync Service
        |
   -----------------
   |       |       |
 EC2-1   EC2-2   EC2-3
```

Every EC2 instance runs `chronyd`, which periodically asks the Amazon Time Sync Service for the correct time.

### Interview question: "How does the NTP server know the correct time?"

The NTP server itself synchronizes with **highly accurate reference clocks**, such as:

- Atomic clocks
- GPS satellites
- National time laboratories

AWS manages this infrastructure for you. You don't need to worry about where the Amazon Time Sync Service gets its time; you simply trust it as the authoritative source.

So the hierarchy is:

```
Atomic Clock / GPS
        ↓
Amazon Time Sync Service (NTP Server)
        ↓
chronyd on EC2
        ↓
Operating System Clock
        ↓
Applications (TLS, JWT, AWS CLI, Kubernetes, etc.)
```

This is the complete chain from the world's most accurate clocks down to your application reading the current time.
