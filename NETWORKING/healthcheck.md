You're thinking in the right direction: a health check is fundamentally "perform some test and interpret the result."

But the exact test depends on the type of health check.

For an AWS Application Load Balancer (ALB), the load balancer does **not** run commands like `ping`, `netstat`, or `top` on your EC2 instance. It has no shell access to the instance.

Instead, the ALB periodically sends network requests to the target.

For example:

```
ALB ----HTTP GET /health----> EC2
```

The application responds:

```
HTTP 200 OK
```

If enough consecutive successful responses are received, the target becomes healthy.

If the ALB gets:

```
HTTP 500
```

or

```
Connection refused
```

or

```
Timeout
```

then it starts marking the target unhealthy.

Conceptually:

```
while True:    response = send_http_request(instance, "/health")    if response.status == 200:        healthy_count += 1    else:        unhealthy_count += 1
```

The health-check request is generated by the load balancer software running on AWS infrastructure, not by code running inside your instance.

Now compare different health checks:

|Health Check Type|What Actually Happens|
|---|---|
|Ping|Send ICMP packet and wait for reply|
|TCP|Try to open TCP connection to port 80/443/8080|
|HTTP|Send HTTP request and verify status code|
|HTTPS|Same as HTTP but over TLS|
|Kubernetes Liveness Probe|Kubelet sends HTTP/TCP/command probe|
|EC2 Status Check|AWS hypervisor checks infrastructure health|

Example of a TCP health check:

```
Health checker     |     | TCP SYN     vInstance:8080
```

If the TCP handshake succeeds:

```
SYN -> SYN/ACK -> ACK
```

the target is healthy.

Notice that no HTTP request is needed. The checker only verifies that something is listening on the port.

Your mention of `netstat` is interesting because **inside the operating system**, `netstat` also checks socket state, but the load balancer doesn't execute `netstat`. Instead, it attempts a real network connection from outside. If nothing is listening, the connection fails and the target becomes unhealthy.

A deeper question is: **What if the application is hung?**

Suppose:

```
Nginx running     -> Port 80 openJava app deadlock -> Requests never complete
```

A TCP health check might still pass because port 80 accepts connections.

An HTTP health check may fail because:

```
GET /health
```

never gets a response before timeout.

This is why mature systems use application-level health endpoints (`/health`, `/ready`, `/live`) instead of just checking whether a port is open.

The important systems-thinking insight is:

**A health check is only as good as what it tests.**

- Ping proves machine reachable.
    
- TCP proves process listening.
    
- HTTP proves application responding.
    
- Custom `/health` endpoint can prove database, cache, and dependencies are working.
    

The closer the health check gets to actual business functionality, the more trustworthy it becomes.
