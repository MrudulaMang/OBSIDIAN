**SHA-256 (Secure Hash Algorithm 256-bit)** is a cryptographic hash function that takes input data of any size and produces a fixed-size **256-bit (32-byte)** hash value, typically represented as a **64-character hexadecimal string**.

It is part of the SHA-2 family, designed by National Security Agency.

Think of **SHA-256 as a fingerprint machine for data.**

Imagine you have a document.

```
My name is Mrudu
```

You put it into a special machine (SHA-256).

The machine gives you:

```
A8F9D3E7C4B6...
```

This long string is called the **hash** or **checksum**.

### Example 1: Your Name

Input:

```
Mrudu
```

SHA-256 machine:

```
   +-------------+   |   SHA-256   |   +-------------+         |         v5b9a7c8d4e...
```

Now suppose you change just **one letter**.

Input:

```
Mrudhu
```

Same machine:

```
   +-------------+   |   SHA-256   |   +-------------+         |         vf91c2e6ab3...
```

Notice the output is **completely different**.

Even a tiny change produces a totally different hash.

---

## Why is this useful?

Suppose your friend sends you a file.

He also tells you:

```
SHA-256:ABC123XYZ...
```

You download the file and run SHA-256 on it.

If you get:

```
ABC123XYZ...
```

✅ The file is exactly the same.

If you get:

```
DEF456LMN...
```

❌ The file has changed (it may be corrupted or tampered with).

---

## Real-life analogy

Imagine every person has a unique fingerprint.

```
Person  →  FingerprintAlice   →  A1B2C3Bob     →  X9Y8Z7John    →  M5N4P3
```

The fingerprint doesn't tell you how to recreate the person.

Similarly,

```
File  →  SHA-256 Hashphoto.jpg  →  4A8C...movie.mp4  →  D1F9...resume.pdf →  98BC...
```

The hash is like the file's fingerprint.

---

## Can we get the original file back from the hash?

No.

Suppose I tell you:

```
Hash:4A8CF91B...
```

Can you tell me what the original file was?

- A photo?
- A movie?
- A PDF?
- A password?

You **can't**. The hash doesn't contain enough information to reconstruct the original data.

---

## Why do DevOps engineers use SHA-256?

Imagine you download Terraform.

```
terraform.zip
```

The website says:

```
SHA-256:A1B2C3D4...
```

You run:

```
sha256sum terraform.zip
```

Output:

```
A1B2C3D4...
```

Since the values match, you know:

- The download completed correctly.
- Nobody modified the file during download.

## One sentence to remember

> **SHA-256 creates a unique digital fingerprint of data. If the data changes even slightly, the fingerprint changes completely.**

### How it works

Input:

```
Hello World
```

Output:

```
a591a6d40bf420404a011733cfb7b190d62c65bf0bcda32b57b277d9ad9f146e
```

Notice that:

- Input can be 1 byte or 1 GB.
- Output is always 256 bits (64 hexadecimal characters).

### Key properties

**1. Deterministic**

The same input always produces the same hash.

```
Hello → 185f8db32271fe25...Hello → 185f8db32271fe25...
```

**2. One-way**

You cannot recover the original input from the hash.

```
Password → Hash
```

Possible.

```
Hash → Password
```

Not feasible.

**3. Avalanche effect**

A tiny change in input completely changes the output.

```
Hello
```

↓

```
185f8db32271fe25...
```

```
hello
```

↓

```
2cf24dba5fb0a30e...
```

Only the capitalization changed, but the hash is entirely different.

**4. Fixed-length output**

Whether the input is:

```
A
```

or

```
A 100 GB file
```

the output is always 256 bits.

### Common uses

**File integrity**

When downloading software, a SHA-256 checksum lets you verify the file wasn't corrupted or altered.

Example:

```
Expected:4f3c2e9d...Downloaded:4f3c2e9d...
```

They match, so the file is intact.

**Password storage**

Applications typically store a hash of the password rather than the password itself.

```
Password    ↓SHA-256    ↓Stored in database
```

In practice, modern systems usually use specialized password-hashing algorithms such as bcrypt, Argon2, or PBKDF2 instead of plain SHA-256, because they are designed to resist brute-force attacks.

**Git object IDs**

Historically, Git used SHA-1 for object IDs. Newer repositories can use SHA-256 for stronger security.

**Blockchain**

Many cryptocurrencies use SHA-256 in mining and transaction verification.

### SHA-256 is not encryption

This is a common interview question.

|Encryption|SHA-256 Hashing|
|---|---|
|Reversible with a key|One-way|
|Used to keep data secret|Used to verify integrity|
|Original data can be recovered|Original data cannot practically be recovered|

Example:

Encryption:

```
Secret   ↓Encrypted   ↓Decrypt   ↓Secret
```

SHA-256:

```
Secret   ↓SHA-256   ↓Hash
```

There is no decryption step.

### Linux commands

Hash a file:

```
sha256sum file.txt
```

Example output:

```
d2d2d2d2d2d2...  file.txt
```

Hash a string:

```
echo -n "hello" | sha256sum
```

### DevOps use cases

- Verifying downloaded binaries (Terraform, Docker, kubectl, etc.).
- Validating container images and artifacts.
- Comparing files to detect changes.
- Generating checksums for backups.
- Ensuring CI/CD artifacts haven't been modified during transfer.

### Interview summary

If asked, "What is SHA-256?" a concise answer is:

> SHA-256 is a one-way cryptographic hash function that converts input data of any size into a fixed 256-bit hash. It is used to verify data integrity, generate checksums, and support security-sensitive systems. Unlike encryption, hashing is not reversible.
