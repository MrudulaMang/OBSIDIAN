# 1. GitHub Actions Architecture

### Workflow

A YAML file stored in:

```
.github/workflows/
```

Example:

```
name: CIon:  push:jobs:  build:    runs-on: ubuntu-latest    steps:      - uses: actions/checkout@v4      - run: echo "Build"
```

### Hierarchy

```
Workflow └── Jobs      └── Steps           └── Actions/Commands
```

### Job

- Runs on a runner.
- Jobs run in parallel by default.
- Use `needs` for dependency.

```
jobs:  build:  test:    needs: build
```

### Step

Single task inside a job.

```
steps:  - run: npm install  - run: npm test
```

### Action

Reusable code.

```
- uses: actions/checkout@v4
```

Interview Question:

**Difference between Action and Step?**

Step = execution unit.

Action = reusable component executed within a step.

# 1. Workflow Architecture

**Q: Explain GitHub Actions Workflow Architecture.**

> GitHub Actions follows a hierarchical structure consisting of Workflows, Jobs, Steps, and Actions.
> 
> A workflow is an automated process defined in a YAML file and stored inside the `.github/workflows` directory. It is triggered by events such as a push, pull request, manual execution, or a schedule.
> 
> Each workflow contains one or more jobs. A job represents a logical unit of work and runs on a runner. Jobs run in parallel by default, but dependencies can be established using the `needs` keyword.
> 
> Within each job, there are multiple steps. A step is an individual task such as checking out code, running tests, or building a Docker image.
> 
> Steps can either execute shell commands directly or invoke reusable actions. Actions are packaged units of reusable automation maintained by GitHub, third parties, or internal teams.
> 
> This architecture promotes modularity, reusability, and maintainability while allowing complex CI/CD pipelines to be broken into smaller manageable components.

**Real Example**

> In our Roboshop project, a workflow was triggered whenever developers pushed code. The workflow contained separate jobs for code checkout, SonarQube analysis, Docker image creation, ECR push, and EKS deployment. This separation made troubleshooting easier because failures could be isolated to specific jobs.

# 2. Self Hosted Runners

**Q: Why would you choose self-hosted runners?**

> GitHub-hosted runners are excellent for general-purpose CI/CD workloads because GitHub provisions and manages the infrastructure. However, many enterprise environments have private resources that are inaccessible from the public internet.
> 
> Self-hosted runners solve this problem by allowing organizations to run GitHub Actions jobs on their own infrastructure such as EC2 instances, Kubernetes clusters, virtual machines, or physical servers.
> 
> This enables workflows to securely access private EKS clusters, internal APIs, private databases, and restricted corporate networks.
> 
> Another advantage is customization. Organizations can preinstall tools such as Terraform, kubectl, Docker, AWS CLI, and security scanners, reducing setup time and improving build performance.
> 
> The tradeoff is operational responsibility. Unlike GitHub-hosted runners, self-hosted runners require patching, monitoring, scaling, and security management.

**Troubleshooting**

> If a runner goes offline, I first verify the runner service status, network connectivity to GitHub, token validity, firewall rules, and disk utilization. Many runner failures are caused by expired registration tokens or infrastructure issues rather than GitHub itself.

---

# 3. OIDC Authentication with AWS

**Q: Why use OIDC instead of AWS access keys?**

> Traditionally, CI/CD pipelines authenticate to AWS using access keys stored as secrets. While functional, this introduces several security challenges including credential rotation, secret management, accidental exposure, and long-lived credentials.
> 
> OpenID Connect (OIDC) provides a more secure approach. GitHub acts as an identity provider and issues a signed token during workflow execution. AWS validates this token against a trust relationship configured in IAM and then issues temporary STS credentials.
> 
> Because credentials are generated dynamically and expire automatically, there is no need to store AWS access keys inside GitHub Secrets.
> 
> This significantly reduces attack surface, simplifies credential management, and aligns with modern cloud security best practices.

**Architecture**

```
GitHub Workflow       ↓OIDC Token       ↓AWS IAM Trust Policy       ↓STS Temporary Credentials       ↓Access AWS Resources
```

**Benefit**

> The biggest benefit is eliminating long-lived credentials from the CI/CD platform.

# 4. Secrets Management

**Q: How do you manage secrets in GitHub Actions?**

> Secrets are sensitive values such as API keys, passwords, certificates, and tokens that should never be hardcoded in source code repositories.
> 
> GitHub provides repository-level, environment-level, and organization-level secrets. During workflow execution, these values can be securely injected into jobs.
> 
> In production environments, I prefer storing only minimal secrets in GitHub and using cloud-native secret management systems such as AWS Secrets Manager, Parameter Store, or HashiCorp Vault.
> 
> This ensures centralized secret rotation, auditing, access control, and compliance.
> 
> Additionally, GitHub automatically masks secret values from workflow logs to reduce accidental exposure.

**Security Consideration**

> Forked pull requests do not receive repository secrets by default, preventing attackers from extracting sensitive information through malicious workflow changes.

# 5. Matrix Strategy

**Q: What is Matrix Strategy and why use it?**

> Matrix strategy allows a single job definition to execute multiple variations of the same workload.
> 
> Instead of writing separate jobs for each operating system, runtime version, or configuration, GitHub dynamically generates jobs based on matrix combinations.
> 
> This improves maintainability while ensuring broader test coverage.
> 
> For example, if an application supports Node.js 18 and Node.js 20, matrix strategy allows testing both versions simultaneously.

**Example**

```

```

executed in parallel.

**Business Value**

> Matrix builds help identify compatibility issues early without duplicating workflow code.

# 6. Reusable Workflows

**Q: Why use reusable workflows?**

> Large organizations often manage hundreds of repositories that require identical CI/CD processes.
> 
> Without reusable workflows, every repository would maintain its own copy of pipeline logic, leading to duplication and maintenance challenges.
> 
> Reusable workflows enable teams to centralize pipeline definitions and invoke them from multiple repositories.
> 
> This improves standardization, governance, and maintainability because updates only need to be made in one place.

**Example**

# 7. Composite Actions

**Q: Difference between Composite Actions and Reusable Workflows?**

> Both promote reusability but operate at different levels.
> 
> A Composite Action bundles multiple steps into a single reusable component.
> 
> A Reusable Workflow can contain multiple jobs, environments, approvals, permissions, and execution logic.
> 
> If I need to reuse a sequence of steps such as checkout, dependency installation, and testing, I use a Composite Action.
> 
> If I need to reuse an entire CI/CD process involving multiple jobs and environments, I use a Reusable Workflow.

**Interview One-Liner**

> Composite Actions reuse steps. Reusable Workflows reuse jobs and pipelines.

# 8. Cache vs Artifacts

**Q: Explain Cache vs Artifact.**

> Although both involve storing files, they serve completely different purposes.
> 
> Caching is a performance optimization mechanism. It stores dependencies such as npm packages, Maven repositories, or Python modules so future workflow executions can reuse them instead of downloading them again.
> 
> Artifacts are workflow outputs. Examples include test reports, coverage reports, deployment packages, and compiled binaries.
> 
> If my goal is reducing pipeline execution time, I use caching.
> 
> If my goal is preserving or sharing generated files, I use artifacts.

**Interview Example**

> Maven dependencies belong in cache. Build.jar belongs in artifact storage.

# 10. Concurrency Control

**Q: Why use concurrency?**

> CI/CD systems frequently receive multiple commits within a short period.
> 
> Without concurrency controls, several deployment workflows may attempt to modify the same environment simultaneously.
> 
> This can lead to race conditions, inconsistent deployments, resource conflicts, and rollback complications.
> 
> Concurrency ensures only one workflow from a specific group executes at a time.
> 
> Depending on configuration, newer executions can either wait or automatically cancel older runs.
> 
> This guarantees deployment consistency and reduces operational risk.

**Real Example**

> If five developers push commits to the main branch within ten minutes, I usually want only the latest deployment to reach production. Concurrency allows older deployment runs to be cancelled automatically.

# Cache Optimization

**Q: How do you optimize GitHub Actions pipeline performance?**

> Pipeline performance directly impacts developer productivity. If every build takes 30-40 minutes, feedback cycles become slow and teams deliver software less efficiently.
> 
> My first focus is dependency caching. Dependencies such as npm packages, Maven repositories, pip libraries, or Gradle caches rarely change between builds, so downloading them repeatedly wastes time.
> 
> GitHub Actions provides cache mechanisms that store dependencies and restore them in future runs using cache keys.
> 
> Beyond caching, I look for opportunities to parallelize jobs. Unit tests, linting, security scans, and build processes often run independently and can execute simultaneously.
> 
> I also review Docker image builds to identify inefficient layer ordering. Frequently changing files should be copied later in the Dockerfile to maximize layer reuse.
> 
> Finally, I analyze workflow logs to identify bottlenecks rather than assuming the problem is infrastructure related.

# Artifacts

**Q: What are artifacts and why are they important?**

> Artifacts are files generated during workflow execution that need to be preserved beyond the lifecycle of a job.
> 
> Common examples include test reports, code coverage reports, deployment packages, Terraform plans, and compiled binaries.
> 
> Since GitHub Actions jobs run in isolated environments, files generated in one job are not automatically available in another.
> 
> Artifacts allow those outputs to be stored and shared across jobs or downloaded later for troubleshooting and auditing purposes.
> 
> In enterprise environments, artifacts also provide traceability because teams can identify exactly which binary was tested, approved, and deployed.

# Environment Approvals

**Q: How do you safely deploy to production?**

> Production environments require stronger controls than development environments because failures directly impact customers.
> 
> GitHub Environments provide approval gates, branch restrictions, required reviewers, and deployment protection rules.
> 
> In a typical workflow, deployments to development and staging occur automatically, but production deployment requires manual approval from authorized reviewers.
> 
> This introduces human validation before high-risk changes reach customers.
> 
> Environment approvals are also valuable for compliance requirements because they create an auditable record showing who approved a deployment and when.

# 2. Secrets Printed in Workflow Logs

**Q: What do you do immediately?**

> I treat this as a security incident.
> 
> My first action is to rotate the exposed credential immediately because I must assume the secret has been compromised.
> 
> Next, I revoke any active sessions or tokens associated with that credential.
> 
> I then restrict access to workflow logs and assess who may have viewed them.
> 
> After containment, I investigate how the secret became exposed. Common causes include debug statements, shell scripts using echo commands, or improper variable handling.
> 
> Finally, I implement safeguards such as secret masking, code reviews, and automated scanning to prevent recurrence.

---

# 3. Self-Hosted Runner Compromised

**Q: What is your response process?**

> I follow standard incident response procedures.
> 
> First, I isolate the compromised runner from the network to prevent lateral movement.
> 
> I immediately stop accepting new jobs on that runner.
> 
> Since CI/CD systems often have elevated privileges, I assume credentials on the runner may be compromised.
> 
> Therefore, I rotate AWS credentials, GitHub tokens, SSH keys, and any other secrets accessible to the runner.
> 
> Next, I collect logs and forensic evidence to understand the attack path.
> 
> Rather than attempting to repair the runner, I typically rebuild it from a trusted image because rebuilding is faster and more reliable than proving a compromised system is clean.
> 
> Finally, I perform a root cause analysis and implement controls to prevent similar incidents.

# 5. Rollback Strategy

**Q: How would you implement rollback?**

> A rollback strategy should be planned before deployment, not after failure occurs.
> 
> In containerized environments, I prefer immutable deployments where each release is tagged with a unique version.
> 
> If a deployment introduces issues, rollback simply means redeploying the previously known-good version.
> 
> In Kubernetes, rollback can be performed using deployment revision history or by redeploying the previous image tag.
> 
> For infrastructure changes managed through Terraform, rollback is more complex because infrastructure state may have changed. In those cases, I prefer controlled forward fixes whenever possible.
> 
> The objective is to minimize recovery time while maintaining system stability.

# 8. Why OIDC Is Better Than Long-Lived Credentials

**Q: Why is OIDC preferred?**

> Long-lived credentials create operational and security challenges.
> 
> They must be stored securely, rotated periodically, and monitored for compromise.
> 
> If exposed, an attacker may retain access until the credentials are manually revoked.
> 
> OIDC replaces static credentials with short-lived, dynamically generated credentials issued at runtime.
> 
> AWS validates the identity of the workflow before granting temporary access.
> 
> This reduces credential exposure, eliminates secret storage requirements, and follows the principle of least privilege.

---

# 9. Prevent Simultaneous Production Deployments

**Q: How do you prevent concurrent deployments?**

> Multiple deployments targeting the same environment can create race conditions and inconsistent application states.
> 
> GitHub Actions provides concurrency groups that ensure only one deployment workflow executes for a given environment at a time.
> 
> If another deployment starts while one is already running, it can either wait or cancel the older execution depending on configuration.
> 
> Combined with environment approvals, this provides controlled and predictable production releases.
