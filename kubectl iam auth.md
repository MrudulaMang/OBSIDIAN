# 🧠 Big Picture

> **EKS does NOT use Kubernetes native users/passwords**  
> It uses **AWS IAM for authentication**

So the flow is:

kubectl → kubeconfig → AWS IAM → EKS API Server → aws-auth → Kubernetes RBAC

---

# 🔥 Step-by-step: What actually happens

## 1. You run a command

kubectl get pods

👉 kubectl looks at your `~/.kube/config`

---

## 2. kubeconfig uses AWS exec plugin

Inside kubeconfig you’ll see something like:

users:  
- name: arn:aws:eks:...  
  user:  
    exec:  
      command: aws  
      args:  
        - eks  
        - get-token

👉 This is the **key piece**

---

## 3. AWS CLI generates a token

Behind the scenes:

aws eks get-token --cluster-name roboshop

This creates a **temporary authentication token** using your IAM identity.

---

## 🧠 What is this token really?

It’s NOT a simple token.

👉 It is a **signed AWS STS request**

Based on:

- your IAM user / role
- access key / session
- signature (SigV4)

---

## 4. Request goes to EKS API server

Now kubectl sends:

GET /api/v1/pods  
Authorization: Bearer <token>

---

## 5. EKS verifies IAM identity

EKS control plane:

1. Validates token signature (SigV4)
2. Calls AWS STS internally
3. Confirms:

"Who is this IAM identity?"

Example:

- `arn:aws:iam::123:user/dev-user`
- `arn:aws:iam::123:role/admin-role`

---

## ⚠️ Important

👉 At this point:

- AWS knows **who you are**
- But Kubernetes still doesn’t know **what you can do**

---

## 6. aws-auth ConfigMap (THE BRIDGE)

Inside cluster:

kubectl -n kube-system get configmap aws-auth -o yaml

Example:

mapRoles: |  
  - rolearn: arn:aws:iam::123:role/admin-role  
    username: admin  
    groups:  
      - system:masters

---

## 🔥 This is the critical mapping

👉 IAM → Kubernetes identity

|IAM|Kubernetes|
|---|---|
|IAM Role/User|username|
|IAM Role/User|groups|

---

## 7. Kubernetes RBAC takes over

Now standard Kubernetes RBAC applies:

ClusterRoleBinding:  
  group: system:masters

👉 This decides:

- Can you list pods?
- Can you delete nodes?
- Can you create deployments?

---

# 🧠 Full flow (clean mental model)

kubectl  
  ↓  
kubeconfig (exec: aws eks get-token)  
  ↓  
AWS IAM (who are you?)  
  ↓  
EKS API server validates token  
  ↓  
aws-auth ConfigMap (map IAM → K8s user/group)  
  ↓  
Kubernetes RBAC (what can you do?)

# ⚡ Real-world debugging scenarios

## ❌ Case 1: “Unauthorized”

👉 IAM auth failed

Check:

- `aws sts get-caller-identity`
- credentials / role

---

## ❌ Case 2: “Forbidden”

👉 IAM works, RBAC failed

Meaning:

- You are authenticated
- But not authorized

Fix:

- Update `aws-auth`
- Add RBAC role binding

---

## ❌ Case 3: Works for admin, not for others

👉 Missing IAM mapping in `aws-auth`

---

# 🔥 Senior-level insight (this is what impresses)

### 1. No static credentials in Kubernetes

- No passwords stored
- Everything is IAM-driven

---

### 2. Token is short-lived

- Usually ~15 minutes

👉 More secure than static kubeconfig certs

---

### 3. aws-auth is a bottleneck

- Manual mapping
- Can break access if misconfigured

👉 Many orgs replace it with:

- OIDC + IAM roles for service accounts (IRSA)

---

# 🧠 One-line interview answer

> “EKS uses IAM for authentication via a token generated by aws eks get-token, which is validated by the API server, then mapped to Kubernetes users via the aws-auth ConfigMap, and finally authorized using Kubernetes RBAC.”
